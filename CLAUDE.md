# ccbot

ccbot ‚Äî Telegram bot that bridges Telegram Forum topics to Claude Code sessions. A topic binds to either a host tmux window (legacy, default) or a docker container (optional, feature-flagged), each running one Claude Code instance.

Tech stack: Python, python-telegram-bot, tmux, uv.

## This repo is PUBLIC ‚Äî never commit private data

Code **and git history** are public. Nothing that identifies the operator or a real user goes in: no secrets/tokens/`.env` (see `~/CLAUDE.md`), no real Telegram user/chat IDs, no personal paths (`/home/<user>/‚Ä¶`, real project names ‚Üí genericize to `/home/user/project`), no real conversation content, no third-party proprietary text (e.g. an agent CLI's full system prompt). **Test fixtures captured from live data MUST be sanitized or rebuilt synthetically** ‚Äî a real transcript embeds paths/prompts in many places; prefer a purpose-built minimal fixture (see `tests/fixtures/codex/rollout-sample.jsonl`). If a leak already landed in a commit, amend/rewrite it out before pushing ‚Äî history is public too.

## Common Commands

```bash
uv run ruff check src/ tests/         # Lint ‚Äî MUST pass before committing
uv run ruff format src/ tests/        # Format ‚Äî auto-fix, then verify with --check
uv run pyright src/ccbot/             # Type check ‚Äî MUST be 0 errors before committing
./scripts/restart.sh                  # Restart the ccbot service after code changes
ccbot hook --install                  # Auto-install Claude Code SessionStart hook
```

## Core Design Constraints

- **1 Topic = 1 Binding = 1 Session** ‚Äî every internal route is keyed by the binding value. Two shapes share that string slot:
  - **Tmux** (default): `@<id>` (`@0`, `@12`) ‚Äî a tmux window on the host. Window name = display name (`window_display_names`); the same directory may have several windows.
  - **Docker** (optional, `DOCKER_AGENTS_ENABLED=true`): `docker:<agent>` ‚Äî Claude Code inside a container, driven via `docker exec <ctn> tmux send-keys/capture-pane -t claude`.
  - `session_manager.resolve_binding(user, thread) ‚Üí ("tmux","@12") | ("docker","assistant") | None` is the transport-selection entry point; `send_to_window` branches internally so most call sites stay transport-agnostic. **Add new transport-agnostic operations as `SessionManager` wrappers** (`send_keys`/`capture_pane`/`kill_agent`/‚Ä¶), not as per-call-site `if docker:` branches.
- **Agent runtime is a SECOND axis, orthogonal to transport** ‚Äî a window runs Claude Code (default), Codex, or Grok, held in `WindowState.runtime` (default `"claude"` ‚Üí zero state migration). Runtime is a window property, **never** in the binding string (`@<id>`/`docker:<agent>` stay transport-only, so `resolve_binding` is untouched). **Put runtime-specific behavior behind `runtimes.AgentRuntime`** (`launch_command`/`iter_transcripts`/`parse_entries`/`exit_command`/`latest_context_tokens`/`is_working`/`has_queued_input`/`panel_actions`/`is_edit_tool`+`diff_header_re` for `/diff`/`native_image_input` for image input/`pane_alive_commands` for the dead-window health check (codex's foreground is `codex`, not `claude`/`node` ‚Äî a hardcoded claude set reaped every codex window 30 s after launch)/`list_sessions`+`picker_icon` for the session picker's agent menu/`history_transcript` for `/commands` history/`interrupt_keys` for `/esc` (codex: Escape only ‚Äî an idle Ctrl-C arms its TUI quit)), NOT `if codex:` at call sites. **Window-bootstrap divergence is CAPABILITIES, never `rt.name != "claude"` comparisons** ‚Äî `uses_session_map` (hook wait + hook-missing alarm + cwd persistence + load_session_map's stale-window sweep; the shared stamp is `session_manager.tag_window_runtime`), `auto_forward_first_message` (codex False: a fresh codex can open on its sign-in menu), `ready_message_key`, `is_available` (`pickable_runtimes()` offers only installed CLIs ‚Äî no codex binary ‚áí no Codex tab ‚áí no dead window typing a missing command); a name comparison silently mis-treats a third runtime as codex-like. Topics remember their runtime (`thread_runtime_memory`, recorded by `record_thread_directory(..., runtime=)`) so the memory rebind after a window death relaunches the SAME CLI ‚Äî `_auto_bind_to_directory` resolves remembered ‚Üí `default_runtime()` (`CCBOT_DEFAULT_RUNTIME`, availability-checked), never hardcodes claude. Hookless runtimes resolve transcripts by cwd ¬´newest wins¬ª, so **two live codex windows on one directory would cross-talk** ‚Äî creation paths refuse the second via `session_manager.has_live_agent_on_cwd` (i18n `bot.same_dir_conflict`); the capped rollout scan has a sticky last-resolved fallback so a quiet window doesn't go dark when its rollout leaves the newest-60 ‚Äî the monitor iterates runtimes and calls the interface (`_process_transcript` is the shared read‚Üíparse‚Üíemit tail); a third model = one subclass, no monitor rewrite. Non-obvious codex bits (all behind the runtime): it has **no SessionStart-hook session_map** (the hook doesn't fire on interactive launch in 0.144.x), so the monitor resolves a codex window's rollout (`~/.codex/sessions/**/rollout-*.jsonl`) by matching `session_meta.cwd` to `WindowState.cwd` (newest wins) and mirrors the rollout's session id onto the window; launch is bare `codex` (no `--name`; resume = `codex resume <id>`; exit = `/quit`; opt-in `CODEX_BYPASS_SANDBOX=true` appends `--dangerously-bypass-approvals-and-sandbox` ‚Äî a **last resort only** (OpenAI doesn't recommend it for personal machines; it's `danger-full-access`). If codex's sandbox can't init (`bwrap: setting up uid map: Permission denied` on Ubuntu 24.04 ‚Äî AppArmor's `bwrap-userns-restrict` only covers `/usr/bin/bwrap`, codex uses its bundled bwrap), the PROPER fix is `apt install bubblewrap` (+ the AppArmor extra-profile), an infra/admin task that keeps codex sandboxed; only then can codex run shell / edit files / read documents. Images work regardless of the sandbox ‚Äî see below); inbound images use codex's **native composer** (`native_image_input`): `session_manager.send_composer_image` types the path so codex attaches it as `[Image #N]` (a CLIENT-side read that bypasses the sandbox), NOT the `(image attached: <path>)` text marker (Claude's way, which codex's sandbox can't open); documents still use the text marker + shell read (so they need `CODEX_BYPASS_SANDBOX` or a working sandbox); its sign-in menu / device-code / onboarding screens reuse the **interactive-UI** (photo + ‚Üë‚Üì‚èé nav + auth-URL surfacing) via a provider-agnostic **`is_login` flag** on `terminal_parser` UI patterns (+`parse_login_code` for the copyable one-time code) ‚Äî Claude's `/login` uses the same path, so **don't build codex-specific login handling**. Busy-detection is runtime-aware: codex has **no `‚îÄ` chrome separator**, so `is_claude_working` can't anchor ‚Äî `CodexRuntime.is_working` matches codex's `(Ns ‚Ä¢ esc to interrupt)` counter (`is_codex_working`) and `has_queued_input` matches its "Messages to be submitted after next tool call" hint. Call sites (restart-guard, `/inject` busy-gate, task-pin idle, reaction-confirm/ack) go through the **`session_manager.is_agent_working`/`agent_has_queued_input`** wrappers (dispatch on `WindowState.runtime` via `window_runtime()`), never `is_claude_working` directly. Codex's approval / pick-an-option menus are caught by a **generic `ChoiceMenu`** UI pattern (cursor on a numbered option `‚Ä∫ N.`/`‚ùØ N.` + a confirm/cancel footer) ‚Äî provider-agnostic, so any new agent's numbered menu surfaces via the same photo+nav path with no new pattern; it's LAST in `UI_PATTERNS` so named patterns (AskUserQuestion's text-surfacing) win first. Non-obvious Grok bits (`GrokRuntime`, probed live on 0.2.111 ‚Äî full detail in its docstring): hookless like codex, but resolution is direct ‚Äî grok's own `~/.grok/active_sessions.json` (live-pid registry) names the cwd's live session, falling back to the newest session dir in `~/.grok/sessions/<url-encoded-cwd>/` (groups matched by DECODING names, so grok's encoder never needs reproducing); the monitored file is **`updates.jsonl` ‚Äî NEVER "simplify" to the flatter `chat_history.jsonl`**: `/compact` rewrites that one in place, which breaks byte-offset incremental reads (updates.jsonl stays append-only through compaction). Grok's Claude-compat scan **does run `~/.claude/settings.json` hooks incl. `ccbot hook`** ‚Äî harmless only because grok's stdin payload is camelCase (`sessionId`) and hook.py reads snake_case; don't "fix" hook.py to accept camelCase, that would let grok sessions pollute `session_map.json`. Its tool-approval prompt is radio-style (`1 (‚óè)` ‚Äî no cursor glyph, so `ChoiceMenu` can't match ‚Üí own `GrokApproval` pattern, driven by the standard ‚Üë‚Üì‚èé keyboard) and its **preselected option is ¬´Yes, and don't ask again¬ª = always-approve**: a blind ‚èé (incl. üëç-to-confirm) flips grok into YOLO mode. No `/diff` (grok draws edit diffs colour-only, no ¬± gutter for the shared crop engine); `/mcps` not `/mcp` (`panel_slash_commands` override); welcome screen has a LIVE composer ‚Üí `auto_forward_first_message=True` (unlike codex).
- **Topic-only** ‚Äî no non-topic mode, no `active_sessions`, no `/list`, no General-topic routing.
- **No truncation at the parse layer** ‚Äî full content is preserved; the only split point is the send layer (`split_message`, 4096-char limit).
- **MarkdownV2 only** ‚Äî user-facing sends go through `safe_reply`/`safe_edit`/`safe_send` (auto-fallback to plain text). Internal queue/UI code calls the bot API directly with its own fallback.
- **Hook-based session tracking** ‚Äî Claude Code's `SessionStart` hook writes `session_map.json`; the monitor polls it to detect session changes. Only sessions present in a session_map source are monitored.
- **User-facing strings are bilingual (ru/en) through `i18n.py` ‚Äî a new one goes through `tr("ns.key")` + a catalog entry, NEVER a hardcoded literal.** The `ru` value is written for a non-technical reader (plain words over jargon: ¬´–∑–∞–≤–µ—Ä—à–∏—Ç—å —Å–µ—Å—Å–∏—é¬ª, not ¬´Kill¬ª); missing keys fall back to `ru`, so a gap degrades, never crashes. Language is one global setting (`/lang`, persisted; default `CCBOT_DEFAULT_LANG`). NOT translated (leave as-is): agent-directed text (voice protocol), logs, and Claude Code's own pane markers matched for parsing. The menu-button `filters.Regex` dispatcher matches **every** language's label via `i18n.all_variants` (built once at startup) and `BotCommand` descriptions come from `build_bot_commands()` ‚Äî so a label change is a catalog edit only, no dispatcher/description drift.
- **`is_claude_working` reads only the status line, not the whole pane** ‚Äî `esc to interrupt` legitimately appears in transcripts, tool output, and this codebase's own source, so a blind `.search()` over the pane marks every such agent permanently busy. It checks the spinner/status line above the chrome separator for the *active* form (`esc to interrupt` or a running `(Ns ¬∑ ‚Ä¶` counter) ‚Äî not the lingering `‚úª Cooked for 12s` done-marker.
- **`(send file: /path)` is the security perimeter against a compromised agent** ‚Äî for docker bindings `session_manager.resolve_agent_file_path` accepts **only** `/workspace/*` (‚Üí `agent.workspace_host_path`); any other container prefix (`/auth/...`, `/root/...`) or `..` traversal is rejected. Tmux bindings pass through verbatim (legacy host paths). Don't loosen this ‚Äî bind-mounted secrets like `/auth/.credentials.json` sit one path away.
- **`/remount` (drive plugin) cascades a `docker restart`** to every active docker agent ‚Äî rclone remount invalidates the FUSE snapshots their bind-mounts hold. Live Claude sessions inside containers are interrupted; that's the intended trade-off vs. stale mounts. The whole rclone stack lives in the `drive` plugin (ccbot-plugins), not core.
- **Inbound media for docker bindings** is saved to `<agent.workspace_host_path>/.inbox/<ts>_<name>` and the marker sent to Claude uses the in-container path `/workspace/.inbox/...` ‚Äî the host-side `ccbot/images/` and `ccbot/files/` dirs aren't bind-mounted into the container. Tmux bindings keep the legacy host-path behavior.
- **PTB HTTP timeouts are deliberately bumped** in `bot.py`'s `Application.builder()` (`read/write=30`, `connect=15`, `media_write=60`) over PTB's stock 5 s ‚Äî on a high-latency link a fresh-start `send_photo` would time out client-side while Telegram had already committed the upload, producing a duplicate photo on the next poll tick. Don't restore the stock defaults.
- **`message_reaction` must stay in `main.py`'s `allowed_updates`** (added there iff `config.reaction_confirm_enabled`, default on) ‚Äî Telegram delivers reaction updates only when the type is explicitly listed *and* the bot is a chat admin; drop it and üëç-to-confirm silently dies. The feature: a üëç (`REACTION_CONFIRM_EMOJI`) on an agent-originated topic message ‚Üí `handlers/reaction_confirm.py` confirms after a `REACTION_CONFIRM_DEBOUNCE_SEC` window (removing the reaction in that window cancels it) ‚Äî Enter on a live interactive prompt, ¬´–¥–∞¬ª typed to an idle agent, nothing if the agent is busy. Resolving the topic needs the `(chat_id, message_id) ‚Üí (user, thread)` LRU in that module (the update carries no `message_thread_id`), populated by `message_queue._process_content_task` and `interactive_ui`.

- **`/inject` is a localhost RCE perimeter ‚Äî never loosen `sanitize_inject_text`** ‚Äî the optional fire-and-forget endpoint (`inject/`, feature-flagged on `CCBOT_INJECT_TOKEN`) types a task into an agent's pane *as a prompt*. It addresses agents by name; `session_manager.resolve_agent_binding(name)` maps it to a binding ‚Äî `docker:<name>` for a docker agent, `@<id>` for a live host tmux window of that name ‚Äî so it drives **both docker and host-tmux** agents. It binds a **unix socket** (`~/.ccbot/run/inject.sock`, `0660` under a `0700` dir), NOT TCP ‚Äî unreachable from containers/other uids; that's the whole point (a local client reaches it as the bot's own user). `inject.core.sanitize_inject_text` is the security shield: it strips control/ESC bytes and **defuses a leading `!` or `/`** by prefixing a space ‚Äî a leading `!` would drop Claude Code's TUI into bash command-mode = shell run *outside* the LLM (for a host-tmux agent that's RCE on the host, not just inside a container); a leading `/` would run a slash command (`/clear`, `/exit`) on the idle prompt instead of landing as text. Codes: 403 `forbidden_agent` (not allowlisted) vs 503 `not_running` (allowlisted but no docker agent and no live window ‚Äî host-tmux agents exist only while running), 409 `busy` (`is_claude_working`/`is_interactive_ui`, never barge a live turn), 503 `unavailable` (resolved but pane/send failed). Allowlist `CCBOT_INJECT_AGENTS`; no token ‚Üí server never starts.

## Code Conventions

- Every `.py` file starts with a module-level docstring: one-sentence summary on the first line, then core responsibilities ‚Äî purpose clear within 10 lines. (The docstrings *are* the module inventory; don't re-list it in CLAUDE.md.)
- Telegram: prefer inline keyboards over reply keyboards; `edit_message_text` for in-place updates; callback data under 64 bytes; `answer_callback_query` for instant feedback.

## Configuration

- Config dir: `~/.ccbot/` by default, override with `CCBOT_DIR`.
- `.env` priority: local `.env` > config-dir `.env`.
- State files: `state.json` (thread bindings, window states, display names, read offsets, voice-mode topics, diff-mode topics, UI language, dashboard message ids), `session_map.json` (hook-generated), `monitor_state.json` (byte offsets).
- `ffmpeg` is required when `GEMINI_API_KEY` is set (Gemini TTS returns raw PCM @ 24 kHz piped through `ffmpeg` ‚Üí OGG/Opus); not needed for ElevenLabs/OpenAI. `voice/startup.check_runtime_dependencies()` logs a WARNING at boot if it's missing.
- Docker agents (convention-over-config): `DOCKER_AGENTS_ENABLED=true` + `DOCKER_AGENTS=name1,name2`. Per-agent paths default to `~/agents/<name>` (workspace) and `~/.local/share/<name>/{claude-home,ipc,session-map.json}`, container `<name>`; override any via `DOCKER_AGENT_<NAME>_{CONTAINER,WORKSPACE,CLAUDE_HOME,IPC,SESSION_MAP,VNC_URL}`. Parser is the pure, unit-tested `config._parse_docker_agents(env, home)`. Each agent's in-container `SessionStart` hook writes its own `session_map.json` to the host via bind-mount, keyed `docker:<agent>` (no prefix to strip ‚Äî the key *is* the binding value); the monitor and `session_manager.load_session_map` merge per-agent maps with the host map on every read. The container must run a tmux session named `claude` (hard-coded in `docker_driver`).
- `SENSITIVE_ENV_VARS` (e.g. `TELEGRAM_BOT_TOKEN`, `GEMINI_API_KEY`) are scrubbed from `os.environ` after `Config.__init__` so they can't leak into Claude Code subprocesses.
- **Portability (this is a public repo): deployment-specific integrations live in PLUGINS, not core.** rclone mounts (`drive` plugin) and the preview fleet + live dashboards (`fleet` plugin) ship in ccbot-plugins; core carries only generic seams: `notifications_chat_id` (voice-budget alerts, plugin pushes), `preview_bin`/`preview_registry_path` (worktree teardown's best-effort `preview down`), `session.live_dashboard_message_ids` state, per-agent `vnc_url`. New host-specific functionality goes into a plugin via the hooks in `plugins.py` (`STRINGS`/`bot_commands`/`register_handlers`/`on_startup`/`on_shutdown`/`status_sections`/`status_buttons`/`callback_dispatch`); a generic knob still gets a `Config` field with a `CCBOT_*` override ‚Äî never a hardcoded host literal in a handler.
- **Optional plugins (`plugins.py`):** heavier deployment-specific integrations live as separate `ccbot.<name>` packages, loaded only when named in `CCBOT_PLUGINS` (comma-separated); the public tree ships none. A plugin optionally exposes `STRINGS` / `bot_commands()` / `register_handlers(app)` / `on_startup(app)` / `on_shutdown()` (all `getattr`-looked-up); a configured-but-absent package is logged and skipped, so the core always runs standalone. Full hook contract in the `plugins.py` docstring.
